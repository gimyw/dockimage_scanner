## 테스트 시나리오

### 시나리오: Cold Start 기반 긴급 복구 (DR — Disaster Recovery)

**시나리오 배경**

운영 중이던 서버가 완전히 초기화되거나 새로운 노드로 교체되어야 하는 상황을 가정합니다. 로컬에 이미지 캐시가 전혀 없는 상태에서 Docker Hub에서 이미지를 처음부터 내려받아 서비스 가능한 상태로 올리기까지 걸리는 시간을 측정합니다.

**적용 실제 시나리오**

| 시나리오 | 설명 |
|---|---|
| Kubernetes 노드 장애 | 기존 노드 이탈 후 새 노드에서 이미지 pull → Pod 기동 |
| 오토스케일 (트래픽 급증) | 신규 인스턴스가 레지스트리에서 이미지 받아 서비스 준비 |
| CI/CD 신규 서버 배포 | 배포 파이프라인이 새 서버에 이미지 pull 후 컨테이너 기동 |
| 장애 후 수동 복구 | 운영자가 서비스 복구를 위해 `docker run` 실행 |

**측정 구간 정의**

```
docker rmi -f (캐시 삭제)
     │
     ▼
[Phase 0] Cold Start 환경 초기화 (캐시 삭제 확인)
     │
     ▼
[Phase 1+2] docker pull ──────────────── pull_ms 측정 시작
  ├─ 레이어 다운로드 (네트워크)
  └─ 레이어 압축 해제 (CPU) ─────────── pull_ms 측정 종료
     │
     ▼
[Phase 3] docker run ──────────────────── ready_ms 측정 시작
  ├─ 컨테이너 프로세스 기동
  ├─ Python 인터프리터 로딩
  ├─ 패키지 import
  ├─ 앱 초기화
  └─ HTTP /ready 200 응답 ────────────── ready_ms 측정 종료
     │
     ▼
Total Time to Ready = pull_ms + ready_ms
```

**판단 기준**

- 30,000ms(30초) 이내 `/ready` 200 응답 시 SUCCESS
- 초과 시 TIMEOUT (RTO 달성 실패)

**측정 스크립트**: `verify_full_lifecycle.sh`

## 6. HTTP 부하테스트 (`hey -n 10000 -c 100`)

`hey -n 10000 -c 100`은 동시 연결 100개를 유지하며 총 10,000개의 HTTP 요청을 보내는 부하 시뮬레이션입니다.

- **RPS (Requests Per Second)**: 서버가 초당 처리한 요청 수. 높을수록 처리 능력이 좋습니다.
- **p95**: 전체 요청 중 빠른 순서로 95%가 이 시간 이내에 응답했음을 의미합니다.
- **p99**: 전체 요청 중 99%가 이 시간 이내에 응답했음을 의미합니다. 극단적인 지연 케이스를 잡아냅니다.

| 케이스 | RPS (원본) | RPS (최적화) | p95 (원본) | p95 (최적화) | p99 (원본) | p99 (최적화) | 에러율 |
|---|---:|---:|---:|---:|---:|---:|---:|
| pre1 | 872.5 | **2,391.5** | 121.5ms | 48.6ms | 129.4ms | 52.5ms | 0% |
| pre2 | 1,252.9 | 1,288.7 | 98.2ms | 91.6ms | 119.0ms | 102.3ms | 0% |
| pre3 | 2,499.5 | 2,392.9 | 45.7ms | 47.4ms | 60.8ms | 57.8ms | 0% |

> **pre1의 RPS 2.7배 향상은 이미지 최적화가 아닌 `flask run` → `gunicorn` 교체 효과입니다.**
> pre2/pre3는 원본/최적화 모두 동일한 서버(uvicorn/gunicorn)를 사용하므로 RPS가 거의 같습니다.
> 이미지 최적화 자체는 런타임 처리 성능에 영향을 주지 않습니다.

### 부하테스트가 유의미한 이유

이미지 경량화가 RPS에 직접 영향을 주지 않는다면 이 테스트는 왜 의미가 있을까요?

**1. 회귀(regression) 검증**

최적화 Dockerfile은 multi-stage 분리, venv 격리, slim 이미지 교체 등 구조 변화를 포함합니다. 이 변화가 실수로 의존성 누락, 환경 변수 누락, 런타임 패키지 제외 등의 문제를 유발할 수 있습니다. 부하테스트는 이런 실수가 실제 요청 처리에서 터지는지 검증합니다. **에러율 0%는 최적화 후에도 서비스가 정상 동작함을 증명합니다.**

**2. 엔트리포인트 변경 효과 검증 (pre1)**

`imgadvisor`는 `flask run`(개발 서버)을 `gunicorn`(운영 서버)으로 자동 교체합니다. `flask run`은 싱글 스레드로 동시 요청을 직렬 처리하지만, `gunicorn`은 멀티 워커로 병렬 처리합니다. pre1의 RPS 872 → 2,392 향상은 이 교체가 실제로 유효함을 수치로 보여줍니다. 이미지 크기 절감과 함께 **운영 환경 적합성 개선까지 이루어진 복합 효과**입니다.

**3. p95/p99 안정성 확인**

이미지 구조 변경 후 특정 조건에서 응답 지연이 발생하는지 확인합니다. pre2/pre3 모두 p95·p99가 원본 대비 소폭 개선되거나 동등 수준을 유지해, 경량화가 응답 안정성에도 영향을 주지 않음을 확인할 수 있습니다.

## 전체 요약

| 항목 | pre1 | pre2 | pre3 |
|---|---|---|---|
| **이미지 크기 절감** | 483→95 MB (80.4%) | 438→70 MB (84.0%) | 422→61 MB (**85.5%**) |
| **Pull 단축률** | 77.5% | **87.2%** | 47.5% |
| **Total DR 단축률** | 76.4% | **85.6%** | 38.5% |
| **원본 Total DR** | 134초 | 106초 | 13초 |
| **최적화 Total DR** | 32초 | 15초 | 8초 |
| **RPS 변화** | 872 → 2,392 (+174%) | 1,253 → 1,289 (+3%) | 2,500 → 2,393 (-4%) |
| **RPS 향상 원인** | gunicorn 교체 | 변화 없음 | 변화 없음 |
| **에러율** | 0% | 0% | 0% |

**핵심 결론**

- 이미지 경량화는 **배포/복구 속도**(Pull + Ready)에 직접적인 효과가 있습니다
- 이미지 경량화 자체는 **런타임 처리 성능(RPS)에는 영향을 주지 않습니다**
- pre1의 RPS 향상은 이미지 최적화와 함께 엔트리포인트(`flask run` → `gunicorn`) 교체가 동시에 적용된 결과입니다
- 모든 케이스에서 에러율 0%로, 경량화 후에도 서비스 안정성에 회귀가 없음을 확인했습니다
