# Python Image Optimization Comparison

This document compares the original bloated Python Dockerfile with the optimized multi-stage output generated by `imgadvisor`.

## Original

```dockerfile
FROM python:3.11

RUN apt-get update && apt-get install -y \
    gcc \
    g++ \
    make \
    cmake \
    build-essential \
    libpq-dev \
    libssl-dev \
    libffi-dev \
    wget \
    curl \
    git \
    vim

WORKDIR /app

COPY . .

RUN pip install flask sqlalchemy psycopg2-binary \
    requests numpy pandas scikit-learn \
    celery redis gunicorn

RUN wget https://github.com/jwilder/dockerize/releases/download/v0.6.1/dockerize-linux-amd64-v0.6.1.tar.gz \
    && tar -C /usr/local/bin -xzvf dockerize-linux-amd64-v0.6.1.tar.gz \
    && rm dockerize-linux-amd64-v0.6.1.tar.gz

RUN rm -rf /var/lib/apt/lists/*

ENV FLASK_APP=app.py
ENV FLASK_ENV=production
ENV PYTHONDONTWRITEBYTECODE=1

EXPOSE 5000
CMD flask run --host=0.0.0.0 --port=5000
```

## Optimized

```dockerfile
# -- builder stage --
FROM python:3.11 AS builder
WORKDIR /app
ENV VIRTUAL_ENV=/opt/venv
ENV PATH="/opt/venv/bin:$PATH"
RUN python -m venv $VIRTUAL_ENV
ENV FLASK_APP=app.py
ENV FLASK_ENV=production
ENV PYTHONDONTWRITEBYTECODE=1
ENV PYTHONUNBUFFERED=1
ENV PIP_NO_CACHE_DIR=1
ENV PIP_DISABLE_PIP_VERSION_CHECK=1
RUN apt-get update && apt-get install -y --no-install-recommends gcc g++ make cmake build-essential libpq-dev libssl-dev libffi-dev wget curl git vim \
    && rm -rf /var/lib/apt/lists/*
COPY . .
RUN pip install --no-cache-dir flask sqlalchemy psycopg2-binary requests numpy pandas scikit-learn celery redis gunicorn
RUN wget https://github.com/jwilder/dockerize/releases/download/v0.6.1/dockerize-linux-amd64-v0.6.1.tar.gz && tar -C /usr/local/bin -xzvf dockerize-linux-amd64-v0.6.1.tar.gz && rm dockerize-linux-amd64-v0.6.1.tar.gz
RUN rm -rf /var/lib/apt/lists/*

# -- runtime stage --
FROM python:3.11-slim
WORKDIR /app
ENV VIRTUAL_ENV=/opt/venv
ENV PATH="/opt/venv/bin:$PATH"
ENV FLASK_APP=app.py
ENV FLASK_ENV=production
ENV PYTHONDONTWRITEBYTECODE=1
ENV PYTHONUNBUFFERED=1
ENV PIP_NO_CACHE_DIR=1
ENV PIP_DISABLE_PIP_VERSION_CHECK=1
COPY --from=builder /opt/venv /opt/venv
COPY --from=builder /usr/local/bin /usr/local/bin
COPY --from=builder /app /app
EXPOSE 5000
CMD ["gunicorn", "-w", "2", "-b", "0.0.0.0:5000", "app:app"]
```

## What Changed

### 1. Single-stage to multi-stage

Before:
- build tools, header packages, and dependency installation all happened in the final runtime image

After:
- the heavy build and install work stays in the `builder` stage
- the final stage only copies runtime assets from the builder

Why it matters:
- compilers and development packages no longer ship in production
- the runtime image is smaller and has a lower attack surface

### 2. Runtime base image reduced

Before:
- `python:3.11`

After:
- `python:3.11-slim`

Why it matters:
- the runtime stage starts from a smaller Debian-based Python image
- this keeps package compatibility safer than switching straight to Alpine

### 3. Python dependencies isolated into a venv

Before:
- Python packages were installed into the image default environment

After:
- dependencies are installed into `/opt/venv`
- the runtime stage copies only `/opt/venv`

Why it matters:
- the runtime stage avoids copying the builder's full Python environment layout
- dependency boundaries are cleaner and more predictable

### 4. apt install normalized

Before:
- `apt-get install -y ...`

After:
- `apt-get install -y --no-install-recommends ...`
- apt lists are removed in the same layer

Why it matters:
- avoids recommended-but-unneeded packages
- reduces layer waste from package index data

### 5. pip install normalized

Before:
- `pip install ...`

After:
- `pip install --no-cache-dir ...`
- `PIP_NO_CACHE_DIR=1`
- `PIP_DISABLE_PIP_VERSION_CHECK=1`

Why it matters:
- avoids pip cache growth inside layers
- removes extra version-check noise during build and runtime

### 6. Python runtime env defaults added

Added:
- `PYTHONUNBUFFERED=1`
- `PIP_NO_CACHE_DIR=1`
- `PIP_DISABLE_PIP_VERSION_CHECK=1`

Already present and preserved:
- `PYTHONDONTWRITEBYTECODE=1`

Why it matters:
- logs flush immediately in containers
- runtime avoids unnecessary `.pyc` and pip cache behavior

### 7. Entrypoint changed from dev server to production server

Before:

```dockerfile
CMD flask run --host=0.0.0.0 --port=5000
```

After:

```dockerfile
CMD ["gunicorn", "-w", "2", "-b", "0.0.0.0:5000", "app:app"]
```

Why it matters:
- `flask run` is a development server, not a production server
- `gunicorn` is the correct production WSGI entrypoint for this case
- explicit workers make runtime behavior more predictable

## Expected Outcome

Compared with the original Dockerfile, the optimized version should typically:
- build a smaller runtime image
- remove compilers and development libraries from production
- reduce package-manager cache waste
- start with a production server instead of Flask dev server
- be safer to push, pull, and deploy

## How to Validate

```bash
imgadvisor recommend -f Dockerfile -o optimized.Dockerfile
imgadvisor validate -f Dockerfile --optimized optimized.Dockerfile
```

If you want to compare build and push manually:

```bash
docker build -f Dockerfile -t 0206pdh/imgadvisor-test:original .
docker build -f optimized.Dockerfile -t 0206pdh/imgadvisor-test:optimized .

docker push 0206pdh/imgadvisor-test:original
docker push 0206pdh/imgadvisor-test:optimized
```
